# ADR-0005: Internationalization, localization & multi-currency

- **Status:** Accepted
- **Date:** 2026-06-28
- **Deciders:** Platform engineering
- **Context docs:** [`PLAN.md`](../../PLAN.md) §5.5, §10, §11 (Phase 7) · [`CLAUDE.md`](../../CLAUDE.md) · [ADR-0001](0001-stack.md)

## Context

Phase 7 includes "i18n/l10n + multi-currency polish" (`PLAN.md` §11): multi-language UI, multi-currency,
and locale-aware dates/numbers (`PLAN.md` §5.5). Payroll is money-heavy and global, so formatting
correctness (per-currency decimals, locale grouping/decimal marks, dates) is the part that matters most.

## Decision

**Localization lives at the view layer; the API stays locale-agnostic.** Per CLAUDE.md ("Time: UTC
everywhere; localize only at the view layer") the API continues to return UTC timestamps, integer minor
units, and ISO-4217 currency codes. The web resolves a locale per request and formats. This keeps the
API/contracts stable, caches sharable, and the formatting logic in one place.

**A pure `@payce/i18n` kernel** (Intl-based, no I/O) holds locale negotiation, locale-aware number/date
formatting, multi-currency money formatting, and a small message-catalog primitive — matching the repo's
pure-kernel pattern and making the tricky bits unit-testable in isolation.

**Multi-currency honours each currency's own minor-unit exponent.** `formatMoney(minor, currency, locale)`
derives the decimal places from `Intl` (the ISO-4217 authority: USD/EUR → 2, JPY → 0, BHD/KWD → 3) rather
than assuming "/100", then formats with `Intl.NumberFormat` currency style. This fixes the class of bug
where zero-decimal currencies render wrong. An unparseable code degrades to the number + raw code instead
of throwing.

**Locale resolution: preference cookie → Accept-Language → default**, threaded through server components as
props rather than via React context. Server components can't share context without a client boundary, and
prop-threading is SSR-correct and explicit; a `LocaleSwitcher` persists the choice to a cookie via a server
action (which re-renders the route, so the change is immediate). The browser never needs an access token.

**A minimal message-catalog primitive**, not a full i18n runtime. `createTranslator(catalog)` does flat
`key → string` lookup with `{var}` interpolation — enough to localize chrome without a heavy dependency.
English + Spanish catalogs cover the app/demo chrome; other supported locales fall back to English copy but
still get locale-correct formatting. Richer needs (plurals/ICU) can adopt `Intl.MessageFormat` behind this
seam later.

## Alternatives considered

- **A full i18n framework (next-intl / i18next / FormatJS runtime).** Deferred: heavier dependency and
  build setup than this slice needs. The kernel + catalog seam leaves the door open to adopt one later
  without changing call sites much.
- **Localizing in the API (per-request `Accept-Language`).** Rejected: violates "localize only at the view
  layer", complicates caching, and couples contracts to presentation. The API stays neutral.
- **Translating all product/marketing copy.** Out of scope: long-form copy is product content in its own
  model; this slice localizes navigational/action chrome + all numeric/temporal/money formatting, which is
  the high-value, clearly-correct part. Full string translation is incremental on top of the catalog.
- **React context for locale.** Rejected for the server-component tree (needs a client provider); explicit
  props are simpler and SSR-safe.

## Consequences

- Money, dates, and numbers render correctly per locale and per currency across the app; switching language
  reformats live (visible in the interactive demo).
- New surfaces localize by resolving the request locale and calling the kernel; new UI strings are added to
  the catalog.
- The demo screens became server-rendered (they read the locale) rather than static — still login-free and
  public, just dynamic.
- Only chrome is translated today; untranslated locales fall back to English copy while formatting stays
  correct — an intentional, incremental starting point.
